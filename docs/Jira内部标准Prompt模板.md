# Jira 内部标准 Prompt 模板

本文档用于沉淀适合公司内部 Jira Server 使用的标准 prompt。

目标：
- 统一团队在 Codex 中调用 Jira skill 的表达方式
- 降低不同人使用时的风格偏差
- 适配公司当前常用项目：`TASK`、`HTEK`、`RDTASK`

## 一、通用填写规范

建议所有 Jira 类 prompt 尽量包含以下信息：

```text
1. 目标 skill
2. 项目 key
3. 任务目标
4. 输入材料
5. 输出要求
6. 是否允许直接创建 / 更新 Jira
7. 是否需要先展示方案再执行
```

推荐固定表达：

```text
请使用 $skill-name 完成以下任务。

项目 key：
任务目标：
输入材料：
输出要求：
执行方式：先展示方案，等我确认后再执行
```

## 二、公司内部标准模板

### 1. 项目状态汇报标准模板

适用 skill：`$jira-server-status-report`

```text
请使用 $jira-server-status-report 生成一份 Jira Server 项目状态报告。

项目 key：<填写项目 key>
时间范围：<例如 最近 7 天 / 最近 1 天 / 最近 30 天>
汇报对象：<例如 项目经理 / 部门负责人 / 研发团队 / 管理层>
输出风格：<简洁摘要 / 详细版>

请重点关注：
1. 已完成事项
2. 当前进行中事项
3. 阻塞项与风险
4. 下一步优先事项

如果需要，请继续读取关键 issue 的详情和评论后再总结。
先输出报告，不要修改 Jira 数据。
```

### 2. 需求拆解标准模板

适用 skill：`$jira-server-spec-to-backlog`

```text
请使用 $jira-server-spec-to-backlog 将下面需求拆解成 Jira Server backlog。

项目 key：<填写项目 key>
需求来源：<例如 PRD / 会议纪要 / 临时需求说明>
需求内容：
<粘贴需求正文>

输出要求：
1. 给出父项建议
2. 给出任务 / 子任务拆解
3. 标明每项建议使用的问题类型
4. 如果项目有必填字段，请一并指出

执行方式：先展示拆解方案，等我确认后再创建 Jira。
```

### 3. 问题分诊标准模板

适用 skill：`$jira-server-triage-issue`

```text
请使用 $jira-server-triage-issue 对下面的问题进行分诊。

项目 key：<填写项目 key>
问题描述：
<粘贴错误现象、复现步骤或用户反馈>

请完成：
1. 提取关键词
2. 搜索相似 issue
3. 判断是重复问题、相关问题、回归问题，还是新问题
4. 给出建议动作

执行方式：先展示分诊结果，不要直接创建 Jira。
```

### 4. 会议纪要转任务标准模板

适用 skill：`$jira-server-meeting-to-tasks`

```text
请使用 $jira-server-meeting-to-tasks 将下面会议纪要整理成 Jira Server 任务。

项目 key：<填写项目 key>
会议纪要：
<粘贴会议纪要>

请完成：
1. 提取行动项
2. 识别负责人
3. 生成任务概要
4. 区分顶层任务和子任务

执行方式：先展示拟创建列表，等我确认后再创建 Jira。
```

## 三、按项目区分的专用 Prompt

## 1. `TASK` 项目专用模板

适用特点：
- 偏公司内部综合任务
- 已验证存在必填字段：`customfield_12800`（验收标准）
- 常用问题类型：`任务`、`子任务`

### 1.1 `TASK` 项目状态汇报

```text
请使用 $jira-server-status-report 生成 `TASK` 项目的周报。

项目 key：TASK
时间范围：最近 7 天
汇报对象：项目经理与部门负责人
输出风格：简洁摘要

请重点输出：
1. 已完成任务
2. 处理中任务
3. 待审核任务
4. 阻塞和风险
5. 下周优先事项

只输出报告，不修改 Jira。
```

### 1.2 `TASK` 项目需求转任务

```text
请使用 $jira-server-spec-to-backlog 将下面需求拆解到 `TASK` 项目。

项目 key：TASK
问题类型优先：任务 / 子任务

需求内容：
<粘贴需求>

注意：
1. 请检查并提醒我 `TASK` 项目的必填字段
2. 创建时要考虑验收标准字段
3. 先给出拆解结果，不要直接创建
```

### 1.3 `TASK` 项目会议纪要转任务

```text
请使用 $jira-server-meeting-to-tasks 将下面纪要转成 `TASK` 项目任务。

项目 key：TASK
会议纪要：
<粘贴纪要>

要求：
1. 优先提取明确可执行的行动项
2. 尝试匹配负责人
3. 如果适合，建议挂成子任务
4. 先展示拟创建结果

注意提醒我哪些任务需要补“验收标准”。
```

### 1.4 `TASK` 项目问题分诊

```text
请使用 $jira-server-triage-issue 对下面问题在 `TASK` 项目中做分诊。

项目 key：TASK
问题描述：
<粘贴问题描述>

请输出：
1. 相似 issue
2. 是否重复
3. 是否建议新建任务
4. 如果新建，建议标题和描述结构

先展示结果，不要直接创建。
```

## 2. `HTEK` 项目专用模板

适用特点：
- 偏主线版本项目
- 常见问题类型：`故障`、`需求`、`任务`、`子任务`
- 更适合区分“需求类工作”和“故障类工作”

### 2.1 `HTEK` 项目需求拆解

```text
请使用 $jira-server-spec-to-backlog 将下面内容拆解到 `HTEK` 项目。

项目 key：HTEK
问题类型优先：
1. 需求：功能类工作
2. 故障：问题修复类工作
3. 任务：实现或协调工作

需求内容：
<粘贴需求>

请先给出：
1. 父项建议
2. 子项建议
3. 每一项建议使用的问题类型
4. 如有必填字段，请指出

先不要直接创建。
```

### 2.2 `HTEK` 项目故障分诊

```text
请使用 $jira-server-triage-issue 对下面问题在 `HTEK` 项目中做分诊。

项目 key：HTEK
问题描述：
<粘贴错误现象、复现步骤、影响范围>

请完成：
1. 搜索相似 issue
2. 判断是否已有 `故障`
3. 如果没有，建议新建为 `故障` 还是 `任务`
4. 输出建议标题和描述结构

先展示判断结果，不要直接创建。
```

### 2.3 `HTEK` 项目版本状态汇报

```text
请使用 $jira-server-status-report 为 `HTEK` 项目生成版本状态汇报。

项目 key：HTEK
时间范围：最近 14 天
汇报对象：项目负责人和研发团队
输出风格：详细版

请重点分析：
1. 已完成需求
2. 当前处理中故障
3. 高优先级未关闭 issue
4. 版本风险点
5. 下一阶段重点

只输出报告，不修改 Jira。
```

### 2.4 `HTEK` 项目会议纪要转任务

```text
请使用 $jira-server-meeting-to-tasks 将下面会议纪要整理到 `HTEK` 项目。

项目 key：HTEK
会议纪要：
<粘贴纪要>

要求：
1. 功能类事项优先建议为 `需求` 或 `任务`
2. 问题修复类事项优先建议为 `故障`
3. 先展示拟创建列表
4. 等我确认后再创建
```

## 3. `RDTASK` 项目专用模板

适用特点：
- 偏研发日常任务
- 常见问题类型：`任务`、`子任务`、`文档撰写任务`、`用例撰写任务`
- 适合研发、文档、测试类内部工作

### 3.1 `RDTASK` 项目研发任务拆解

```text
请使用 $jira-server-spec-to-backlog 将下面研发需求拆解到 `RDTASK` 项目。

项目 key：RDTASK
问题类型优先：
1. 任务：研发实现
2. 文档撰写任务：文档整理
3. 用例撰写任务：测试或用例补充
4. 子任务：更细粒度执行项

需求内容：
<粘贴需求>

请输出：
1. 父项建议
2. 子项拆解
3. 每项建议的问题类型

先展示方案，不要直接创建。
```

### 3.2 `RDTASK` 项目技术调研/文档任务

```text
请使用 $jira-server-spec-to-backlog 将下面工作内容拆解到 `RDTASK` 项目。

项目 key：RDTASK
工作内容：
<粘贴技术调研、文档编写、测试设计内容>

要求：
1. 研发实现归类为 `任务`
2. 文档整理归类为 `文档撰写任务`
3. 测试设计归类为 `用例撰写任务`
4. 如果需要，补充建议子任务
```

### 3.3 `RDTASK` 项目日常状态汇报

```text
输出风格：简洁但包含 issue key

请重点关注：

3. 文档和测试任务进度
```

## 四、团队使用建议

### 1. 默认采用“两阶段模式”

推荐固定流程：

```text
第一阶段：先分析、先展示
第二阶段：确认后再创建或更新 Jira
```

这样可以减少误建单、错建类型、漏填必填字段等问题。

### 2. 先检查 issue type 和必填字段

尤其是：
- `TASK` 项目：重点关注验收标准
- `HTEK` 项目：重点确认 `故障 / 需求 / 任务` 的选择
- `RDTASK` 项目：重点区分研发任务、文档任务、用例任务

### 3. 问题分诊默认不直接建单

推荐默认 prompt 里都写上：

```text
先展示分诊结果，不要直接创建 Jira。
```

### 4. 会议纪要默认先展示任务清单

推荐默认写法：

```text
先展示拟创建任务列表，等我确认后再执行。
```

这样最适合公司内多角色协作场景。
