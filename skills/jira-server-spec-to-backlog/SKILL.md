# Jira Server 需求转 Backlog

## 概述

将需求文档转换为可执行的 Jira Server backlog：先选择目标项目，识别项目可用的问题类型，先展示拆解方案，再创建父任务和子任务项。

## 工作流程

1. 获取需求来源。  
接受粘贴文本、Markdown 文档或简要需求描述。

2. 确认目标 Jira 项目 key。  
如果不明确，使用 `list_jira_projects` 辅助选择。

3. 识别项目的问题模型。  
使用：
- `get_jira_issue_types`
- `get_jira_issue_type_fields`

重点检查：
- 项目中是否存在 `Epic`、`需求`、`任务`，还是只有通用问题类型
- 创建时有哪些必填自定义字段

4. 将需求拆为：
- 一个代表整体范围的父级事项
- 若干个可交付的任务
- 在确实需要细分时，再创建子任务

5. 创建前先展示拆解结果。  
展示内容包括：
- 父级事项概要
- 子事项概要
- 每个事项将使用的问题类型

6. 先创建父级事项。  
使用 `create_jira_issue`。  
如果项目支持 `Epic`，优先使用 `Epic`。  
如果不支持，则使用最合适的顶层类型，例如 `需求` 或 `任务`。

7. 再创建子事项。  
兄弟级工作项使用 `create_jira_issue`。  
如果方案明确需要挂在父任务下面，则使用 `create_jira_subtask`。

## 问题类型选择

优先顺序建议如下：
- `Epic`：作为总括性的 backlog 父项
- `需求`：用于功能或需求级工作
- `故障`：用于缺陷修复类工作
- `任务`：用于实现、协调或支撑工作
- 项目自定义的子任务类型：用于更细分的执行项

如果指定类型不存在，就退回到 `get_jira_issue_types` 返回的最接近类型。

## 必填字段处理

创建前先用 `get_jira_issue_type_fields` 检查必填字段。  
如果项目要求自定义字段，优先向用户收集字段值；只有在用户已经确认执行时，才允许填入明显安全的占位值。

## 输出格式

建议先按这个形式展示拆解方案：

```text
父项：
- [问题类型] 概要

子项：
1. [问题类型] 概要
2. [问题类型] 概要
3. [问题类型] 概要
```

创建完成后返回：
- 创建出的 issue key
- 对应 browse URL
- 仍需人工补充的字段

## 注意事项

- 除非用户明确要求立即执行，否则不要在展示拆解前直接创建 issue。
- 任务粒度要实用，避免把一个小功能拆成过多单子。
- 优先复用项目现有的问题类型，不要强行套用 Cloud 风格的字段模型。
