# Jira Server 会议纪要转任务

## 概述

把会议纪要解析成行动项，识别可能的负责人，先向用户确认拟创建的任务列表，再在指定项目中创建 Jira Server 任务或子任务。

## 工作流程

1. 读取会议内容。  
接受粘贴的会议纪要，或整理过的摘要。重点关注明确的行动项、责任人和截止信息。

2. 提取行动项。  
优先识别以下模式：
- `@某人 去做 X`
- `某人 会做 X`
- `TODO: X（某人）`
- 明确包含“谁负责什么”的 bullet 项

3. 确定目标项目。  
必要时使用 `list_jira_projects` 帮助选择。

4. 解析负责人。  
使用 `search_jira_users` 按姓名片段、用户名或邮箱片段查找。  
如果匹配结果不唯一，创建前应先确认。

5. 决定 Jira 结构。  
使用：
- 顶层任务：适用于彼此独立的行动项
- `create_jira_subtask`：适用于明确属于某个已有父任务的跟进事项

创建前应使用 `get_jira_issue_type_fields` 检查必填字段。

6. 先展示解析结果。  
展示内容包括：
- 任务概要
- 建议负责人
- 计划使用的问题类型
- 该项是任务还是子任务

7. 再创建任务。  
使用：
- `create_jira_issue`
- `create_jira_subtask`
- 如果需要后补负责人，可使用 `update_jira_assignee`

## 概要写法

优先使用短句和祈使表达：
- `整理耳机固件更新说明`
- `评审新手引导流程设计稿`
- `确认 OEM 报价单`

任务描述里应补充上下文、原始纪要引用和会议来源。

## 注意事项

- 在提取结果不清晰前，不要直接创建 ticket。
- 一条行动项对应一个可执行交付物。
- 如果负责人身份不明确，优先保持未指派，而不是猜测归属。
